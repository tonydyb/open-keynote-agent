# Proposal: Add Image Assets To Storybook Renderer

## Summary

Connect generated storybook image assets to the existing Keynote storybook renderer.

This change lets users render:

```text
DeckSpec + image_manifest.json/assets -> Keynote storybook with inserted illustrations -> optional PDF
```

It does not generate images. It consumes assets generated by 010/011.

## Motivation

009 can render a deterministic Keynote storybook using text, emoji, and rectangle decorations.

010/011 can generate per-slide PNG illustration assets from `DeckSpec`.

The product flow is still incomplete because the generated images are not inserted into Keynote. Users can generate stories and images, but cannot yet produce an editable Keynote/PDF storybook that includes those images.

012 closes that loop.

## Goals

- Add a `keynote.add_image` tool that inserts a local image file into a Keynote slide.
- Track image objects in the existing object registry so users can later move/resize them.
- Load `image_manifest.json` and resolve portable relative asset paths.
- Extend `render_storybook_deck(...)` to optionally use image assets.
- Extend `oka render-storybook` with an `--images <image_manifest.json>` option.
- Keep fallback behavior: if no image manifest is supplied, current emoji/shape rendering still works.
- Preserve deterministic layout templates and avoid LLM-driven coordinates.
- Keep tests independent of real Keynote unless integration tests are explicitly enabled.

## Non-Goals

- Do not generate images.
- Do not call an LLM.
- Do not improve image prompt quality.
- Do not add OpenAI image provider support.
- Do not build print-ready PDF layout, bleed, trim, or CMYK handling.
- Do not implement advanced image cropping/masking.
- Do not replace Keynote rendering with PPTX/ReportLab.
- Do not perform visual QA on generated slides.

## User Story

Given an already planned deck:

```bash
uv run oka deck-plan "Create a 12-page Cinderella children's storybook..." \
  --slides 12 \
  --output /tmp/cinderella-plan
```

Generate preview illustrations:

```bash
uv run oka generate-images /tmp/cinderella-plan/deck_spec_en.json \
  --provider bedrock \
  --slides 1,4,9 \
  --style soft_storybook_watercolor \
  --output /tmp/cinderella-art-preview
```

Render Keynote with generated images:

```bash
uv run oka render-storybook /tmp/cinderella-plan/deck_spec.json \
  --images /tmp/cinderella-art-preview/image_manifest.json \
  --output /tmp/cinderella-keynote
```

The renderer inserts available slide images into matching slides. If an image is missing for a slide, the renderer keeps the emoji/shape fallback visual for that slide.

## Success Criteria

- `keynote.add_image` inserts an image into a slide and stores local object metadata.
- `render-storybook --images <manifest>` inserts image assets for matching slide indexes.
- Missing selected-slide preview assets do not fail the whole render by default; those slides use existing fallback visuals.
- Invalid manifest paths fail early with clear errors.
- `tool_results.jsonl` includes image insertion tool results.
- The existing no-image renderer behavior remains compatible.
- Unit tests use fake runners/tools and do not require Keynote.
- Real Keynote integration remains opt-in with `RUN_KEYNOTE_INTEGRATION=1`.
